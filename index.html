HTTP3(quic) 直连黑名单域名测试

## HTTP3(quic) 直连黑名单域名测试

根据揭示并绕过中国防火长城基于SNI的QUIC封锁机制,我的看法:
目前(2025年8月)

  • 对quic的封锁漏洞百出,可以轻松绕过
  • 虽然少量流量瘫痪gfw是gfw的bug可以被修复,但审查quic连接的计算量还是比http1/2大不少,这是quic特性无法解决
  • gfw审查不过来quic的时候会直接放行
  • 虽然src_port < dst_port时gfw会直接放行但使用< 443的端口需要提权,不合适

# http3中国直连黑名单域名测试方法

curl测试 (ubuntu curl 8.5.0)

curl --connect-to example.com:443:[x:x:x:x::x]:443 https://example.com

quic测试 (quic-go 0.54.0)

./quic-dl --connect-to [x:x:x:x::x]:443 https://example.com index.html
  • 所有网站使用dns.google取得无污染解析,用无污染的ipv6直连
  • 套Fastly CDN的网站都使用www.fastly.com解析的ipv6地址连接(除twitch.tv报错),所以pinterest.comdns查不到ipv6也是用ipv6连的
  • quic建立连接前先用本机同端口发送两个80-120 bytes的随机数据包干扰审查
  • 不确定这些quic成功连接的网站是
    1.不在quic sni的黑名单里
    2.通过先发送两个随机数据包绕过了gfw的sni审查
    3.gfw负载太大直接放行

Gcore CDN域前置翻墙

# 域前置

客户端向CDN发起https请求，连接CDN时使用假sni，cdn解开https的加密后发现host和sni不一样，就将请求转发给host域名的源服务器（而不是报错或转发给sni的源服务器），大部分CDN服务商不支持这种操作。

# 配置

不良林的教程 不良林的视频

  • 除了tls in tls特征能检测出你在翻墙之外，ip和域名都不是自己的，防火墙封ip封域名不会对翻墙用户造成损失
  • Gcore暂时不会验证host域名的所有者，所以不需要自己有域名
  • 没有域名的话源服务器仅支持ipv4地址，可以自定义端口
  • 每月免费1TB流量，需要绑定支付方式
  • 同样的CDN配置除了vmess+ws也可以过vless+xhttp（vless没有加密，cdn可以看到明文的dns请求和sni）
  • 如果已有一个vmess/vless+ws/xhttp过CDN的xray配置，不需要任何修改就能用
  • Gcore的ip被封锁很严重，即使优选ip也不稳定

# llama-swap

llama-server和llama-swap打包进同一个docker，通过启动多个llama-server进程切换gguf模型，提供一个简单的webui，可以手动加载/卸载模型，也可以在网页上对话

由于llama-cpp没有提供Linux的ROCM二进制文件，只能使用Vulkan，单卡性能和ROCM基本没有区别

Vulkan可以在多张不同品牌的GPU上运行同一个模型，实测一张AMD独显+一张Intel独显能运行，性能可以接受 (16GB+16GB，qwq:32b Q4_K_M 15 token/s)

Cloudflare Zerotrust 使用API删除Access

# 个人用户可能用到的ZeroTrust功能

1.Tunnel:免费内网穿透(需要域名)
2.Access:保护内网穿透的NAS和/自用面板/管理后台等(免费但需绑定支付方式)
3.ZeroTrust VPN(被墙了)

Access很好用,没有通过Cloudflare验证的流量根本不会发到自己的服务器上,也许是最安全的鉴权方式.

Access不绑定支付方式也能添加,但在添加之后在网页上没法修改/删除,会导致自己被锁在网页外面.

以下是用Cloudflare API删除Access的方法

# Linux Curl调用Cloudflare API

Cloudflare Api Document

列出所有Application

curl https://api.cloudflare.com/client/v4/accounts/$CloudflareAccountID/access/apps -H "X-Auth-Email: $Cloudflare Email" -H "X-Auth-Key: $CloudflareApi Key"

删除指定Application

curl https://api.cloudflare.com/client/v4/accounts/$CloudflareAccountID/access/apps/$AppID -X DELETE -H "X-Auth-Email: $Cloudflare Email" -H "X-Auth-Key: $Cloudflare Api Key"

清除bash记录

history -c
rm ~/.bash_history

#

理论上用Cloudflare API也可以在没有绑定支付方式的情况下添加/修改Access,达到正常使用的目的.
但不推荐这样做,如果API突然增加限制就会彻底没法修改了.(没付钱Cloudflare人工客服肯定不会理你,删除账号同一个邮箱不能创建新的).

Mihomo (Clash Meta) API示例

Mihomo官方文档写的很简略,找示例可以在MedaCubeXD的源码里找

也可以在浏览器的F12里监视MetaCubeXD页面的Network,抄到curl里就能用

# 比如

切换节点

curl -H 'Authorization: Bearer API密码' http://192.168.1.1:9090/proxies/节点所在的组/?force=true -d '{"name":"要切换到的节点名称"}' -X PUT

打断所有连接

curl -H 'Authorization: Bearer API密码' http://192.168.1.1:9090/connections -X DELETE

测试某个节点延迟

curl -H 'Authorization: Bearer API密码' 'http://192.168.1.1:9090/proxies/节点名称/delay?url=https://gstatic.com/generate_204&timeout=5000' -X GET

获取所有节点信息

curl -H 'Authorization: Bearer API密码' 'http://192.168.1.1:9090/proxies' -X GET

这些curl命令可以在openwrt的crontab里直接使用

凌晨三点切换节点

0 3 * * * curl -H 'Authorization: Bearer API密码' http://127.0.0.1:9090/proxies/Youtube/?force=true -d '{"name":"HK-xxx"}' -X PUT

Fedora+Windows双系统启动配置

# 1. Fedora 自动删除旧内核

sudo nano /etc/dnf/dnf.conf

保留最新的2个内核

[main]
installonly_limit=2

# 2. GRUB2 设置默认启动分区

sudo nano /etc/default/grub

三种模式

  • 记忆上次选择的启动项
GRUB_DEFAULT=saved
GRUB_SAVEDEFAULT=true
  • 选择第3个启动项
GRUB_DEFAULT=2
  • 选择该名称的启动项
GRUB_DEFAULT="Windows Boot Manager (on /dev/sda1)"

保存

sudo grub2-mkconfig -o /boot/grub2/grub.cfg
