## 使用自签证书的安全提示

  1. 此时hysteria无法防御主动探测,因为无法提供伪装网站的合法ssl证书.
    所以想精准识别hysteria非常容易,现在不封只是GFW对quic的封锁还不成熟.
  2. 客户端建议pin自签的证书,否则可能被中间人攻击

## 性能

使用bbr时跑100mbps对性能没有什么要求

## 速度测试

正常情况下quic直连墙外会被阻断到无法使用.

比如dash.cloudflare.com,F12->Network可以看到大量使用h3下载的js,速度极慢导致网页全天完全不可用,只有浏览器/路由器禁用h3才能正常登录.

但是某垃圾VPS晚高峰测速,hysteria仅用bbr就能100+mbps,同ip同端口的reality被限速0.5mbps.加上dash.cloudflare.comh2正常h3不可用,说明过墙的h3不是拥堵而是是被运营商/GFW主动劣化到不能用的.

我测试用的hysteria可能因为sni是bing.com在白名单里所以不被限速/丢包

## Tips

  • 如果用别人的域名,因为不防主动探测,建议使用ipv6部署hysteria,封了换ip比较容易.
  • GFW审查quic的成本高于其它基于tcp的翻墙协议,因为quic里传输的所有内容都是加密的,从加密的quic中解密出sni的计算量比直接从tcp中读取明文sni大.
  • 使用docker -p [::]:443:443/udp即可仅让此容器监听ipv6发来的udp数据包,同时另一个容器监听同端口的tcp连接-p 443:443/tcp.

## 部署

# 服务端

安装docker

创建一个hysteria目录

mkdir hysteria && cd hysteria

生成自签证书

openssl req -x509 -nodes -newkey ec:<(openssl ecparam -name prime256v1) -keyout ./server.key -out ./server.crt -subj "/CN=bing.com" -days 36500

生成配置文件

nano hysteria.yaml

tls:
  cert: /etc/hysteria/server.crt
  key: /etc/hysteria/server.key

auth:
  type: password
  password: 123456
  
masquerade:
  type: proxy
  proxy:
    url: https://bing.com
    rewriteHost: true

运行&开机自启动

docker run -d --name hysteria --pull always --restart always -p 443:443/udp -v ./hysteria/:/etc/hysteria/ tobyxdd/hysteria server -c /etc/hysteria/hysteria.yaml

检查hysteria是否正常运行

docker logs hysteria

# 客户端

  • 长期使用建议直接用sing-box或mihomo

Windows:下载hysteria-windows-amd64-avx.exe

config.yaml,保存到exe同目录

server: [your-ip]:[your-port]

auth: [your-password]
tls:
  insecure: true

http:
  listen: 127.0.0.1:8080 
  disableUDP: false

我的Windows无法开启Socks5代理,只用http

  • 手动将系统代理设置为127.0.0.1:8080
  • 或Firefox里单独给浏览器设置代理
