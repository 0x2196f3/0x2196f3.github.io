给套Cloudflare CDN的节点开启ECH

## 参考

[5] 冲浪小本本儿 (二)(2-2)—-http-https-ECH 最广泛、最真实、最管用的互联网隐私保护[3]— ECH详谈

## 省流

  • 启用ECH的https连接从外观上(从ISP/GFW的角度观测)除了sni是某ech域名之外和明文client hello的https连接没有明显的区别
  • 由于启用ECH时任何到Cloudflare CDN的https连接sni都是cloudflare-ech.com,GFW阻断该域名Cloudflare的ECH就不能用了
  • 对于翻墙软件,只要有正确的ECH Config即可建立ECH连接,不论这个ECH Config原本是给哪个域名的,是提前配置的固定值/明文DNS查询到的/DoH查询到的
  • 使用Cloudflare CDN的所有网站在同一时刻共用同一个ECH Config,但ECH Config几小时就会换一个
    这代表:
    1.Cloudflare CDN上任何网站的ECH Config都可以用于连接任意使用Cloudflare CDN的域名(即使该域名查不到ECH Config也可以强制ECH连接)
    2.即使一个域名被DNS污染+SNI阻断,拿到Cloudflare CDN的ip和ECH Config即可使用cloudflare-ech.com的sni连接cloudflare的ip访问该网站
    3.不能将事先查好的ECH Config写在翻墙软件的配置文件里,几小时就过期了,需要每次连接时向DNS服务器查询
  • 目前国内存在能返回Cloudflare的正确ECH Config的明文DNS或DoH服务器,但以后可能全部被污染.最好自建一个私人DoH服务器

## 客户端配置

对于套Cloudflare CDN的节点,ECH由Cloudflare的边缘服务器处理,自己的翻墙服务器和Cloudflare面板上不需要任何配置:ECH默认启用且免费账号不能禁用

# Xray客户端

文档
支持编辑ECH的GUI目前基本没有,需要直接改json

在tlsSettings里添加

"tlsSettings": {
    "echConfigList": "cloudflare-ech.com+https://dns.alidns.com/dns-query",
    "echForceQuery": "full"
}
  • full:无法获取ECH Config就断开连接(推荐)
  • half/none:无法获取ECH Config时回退到明文Client Hello,会暴露域名

或在分享链接里添加(大多数GUI目前还不支持)

&ech=cloudflare-ech.com%2Bhttps%3A%2F%2Fdns.alidns.com%2Fdns-query&

cloudflare-ech.com+https://dns.alidns.com/dns-query的意思?

向dns.alidns.com这个DoH服务器发起HTTPS(65)查询,域名是cloudflare-ech.com,使用DoH服务器返回的(原本是给cloudflare-ech.com用的)ECH Config对你节点的ip和域名建立ECH连接

  1. 此时hysteria无法防御主动探测,因为无法提供伪装网站的合法ssl证书.

## 性能

## 速度测试

正常情况下quic直连墙外会被阻断到无法使用.

比如dash.cloudflare.com,F12->Network可以看到大量使用h3下载的js,速度极慢导致网页全天完全不可用,只有浏览器/路由器禁用h3才能正常登录.

但是某垃圾VPS晚高峰测速,hysteria仅用bbr就能100+mbps

可能因为sni=bing.com在白名单里所以没有被限速/丢包

同ip同端口的reality(同ASN的小众域名)被限速0.5mbps,说明过墙的h3不是拥堵而是是被ISP/GFW主动劣化到不能用的

## Tips

  • 如果用别人的域名,因为不防主动探测,建议使用ipv6部署hysteria,封了换ip比较容易.

  • GFW审查quic的成本高于其它基于tcp的协议,因为quic里传输的所有内容都是加密的,从加密的quic中解密出client hello里的sni的计算量比直接从tcp中读取明文sni大.

  • 使用docker -p [::]:443:443/udp即可仅让此容器监听ipv6发来的udp数据包,同时另一个容器监听同端口的tcp连接-p 443:443/tcp.达成hysteria和xray在同ip的443端口共存
    相同的[ip+port],udp和tcp的sni不同也可能是特征

  • 即使使用bbr+真实域名,hysteria也不能套CDN,想套CDN并使用quic过墙可以尝试xray的xhttp

# 连接SSH

# 安装软件包

opkg update
opkg install block-mount kmod-fs-ext4 e2fsprogs parted kmod-usb-storage

# 查看USB设备位置

ls -l /sys/block

如果是无线路由器只有一个外接存储，大概率是sda

lrwxrwxrwx    1 root     root             0 xxx xx xx:xx sda -> ../devices/xxx/block/sda

# 在USB上创建分区，格式化，设置exroot并自动挂载

#Partitioning and formatting
DISK="/dev/sda"
parted -s ${DISK} -- mklabel gpt mkpart extroot 2048s -2048s
DEVICE="${DISK}1"
mkfs.ext4 -L extroot ${DEVICE}

#Configuring extroot
eval $(block info ${DEVICE} | grep -o -e 'UUID="\S*"')
eval $(block info | grep -o -e 'MOUNT="\S*/overlay"')
uci -q delete fstab.extroot
uci set fstab.extroot="mount"
uci set fstab.extroot.uuid="${UUID}"
uci set fstab.extroot.target="${MOUNT}"
uci commit fstab

#Configuring rootfs_data / ubifs
ORIG="$(block info | sed -n -e '/MOUNT="\S*\/overlay"/s/:\s.*$//p')"
uci -q delete fstab.rwm
uci set fstab.rwm="mount"
uci set fstab.rwm.device="${ORIG}"
uci set fstab.rwm.target="/rwm"
uci commit fstab

## HTTP3(quic) 直连黑名单域名测试

根据揭示并绕过中国防火长城基于SNI的QUIC封锁机制,我的看法:
目前(2025年8月)

  • 对quic的封锁漏洞百出,可以轻松绕过
  • 虽然少量流量瘫痪gfw是gfw的bug可以被修复,但审查quic连接的计算量还是比http1/2大不少,这是quic特性无法解决
  • gfw审查不过来quic的时候会直接放行
  • 虽然src_port < dst_port时gfw会直接放行但使用< 443的端口需要提权,不合适

# http3中国直连黑名单域名测试方法

curl测试 (ubuntu curl 8.5.0)

curl --connect-to example.com:443:[x:x:x:x::x]:443 https://example.com

quic测试 (quic-go 0.54.0)

./quic-dl --connect-to [x:x:x:x::x]:443 https://example.com index.html
  • 所有网站使用dns.google取得无污染解析,用无污染的ipv6直连
  • 套Fastly CDN的网站都使用www.fastly.com解析的ipv6地址连接(除twitch.tv报错),所以pinterest.comdns查不到ipv6也是用ipv6连的
  • quic建立连接前先用本机同端口发送两个80-120 bytes的随机数据包干扰审查
  • 不确定这些quic成功连接的网站是
    1.不在quic sni的黑名单里
    2.通过先发送两个随机数据包绕过了gfw的sni审查
    3.gfw负载太大直接放行

Gcore CDN域前置翻墙

# 域前置

客户端向CDN发起https请求，连接CDN时使用假sni，cdn解开https的加密后发现host和sni不一样，就将请求转发给host域名的源服务器（而不是报错或转发给sni的源服务器），大部分CDN服务商不支持这种操作。

# 配置

不良林的教程 不良林的视频

  • 除了tls in tls特征能检测出你在翻墙之外，ip和域名都不是自己的，防火墙封ip封域名不会对翻墙用户造成损失
  • Gcore暂时不会验证host域名的所有者，所以不需要自己有域名
  • 没有域名的话源服务器仅支持ipv4地址，可以自定义端口
  • 每月免费1TB流量，需要绑定支付方式
  • 同样的CDN配置除了vmess+ws也可以过vless+xhttp（vless没有加密，cdn可以看到明文的dns请求和sni）
  • 如果已有一个vmess/vless+ws/xhttp过CDN的xray配置，不需要任何修改就能用
  • Gcore的ip被封锁很严重，即使优选ip也不稳定

# 搜索同ASN同地区可用域名

search.censys.io

搜索框输入以下内容,不登录只能看一页,Per Page选100,Virtual Hosts 选INCLUDE或ONLY

把ASN换成服务器所在ASN（5位数字）， country换成服务器所在国家

autonomous_system.asn:00000 and location.country: "United States" and services: (port: 443 and service_name: HTTP and http.response.status_code: 200 and tls.version_selected: TLSv1_3)

censys search语法

# 检查可用性

  • 不翻墙可以正常访问
  • 网页有内容
  • 路径/使用http/2 (F12 -> Network)
  • 路径/返回200 (F12 -> Network)
  • tls v1.3且支持x25519 (F12 -> Security)

# 检查网站没有使用CDN

  1. 在dns.google查询域名解析出的ip
  2. 拿查到的ip去ip.sb或ip138.com查询ASN归属,确认不属于CDN提供商
  • https://target.domain/cdn-cgi/trace 访问这个路径,有内容说明使用了Cloudflare CDN,有人扫,绝对不能用
  • 一般小网站但解析出多个ip地址很有可能使用了CDN,不能用

# 网站安全检查

  • 在搜索引擎里搜索网站名称,能搜到这个域名说明至少搜索引擎认为这个网站比较正常.
  • 不要用中文网站尤其是有备案号的(有备案号ip肯定在中国),极有可能是黑产网站隐藏在看起来正常的静态网页里.
  • 外国大公司能产生大量流量的网站,要么在中国有CDN要么被墙了,所以最好都不要用.

# 验证伪装域名正常工作

在没有翻墙的电脑改host(被偷的域名 -> VPS ip),用浏览器访问https://域名,没有证书报错并且显示的内容和没改host时一模一样,说明reality正常工作.

搬瓦工VPS配置自定义IPv6地址

搬瓦工提供的ipv6是通过6in4隧道连接到另一台服务器的ipv6出口的,公网ipv6地址在ipv6net上,不是eth0

  • ipv6直连中国速度比ipv4慢不少
  • 删了ipv6重新分配还是原来的那个
  • 可以先用ipv6测试协议会不会被墙，没事再换成ipv4

搬瓦工后台通过ipv6net这个service自动配置6in4隧道,配置脚本在/etc/init_ipv6_tunnel,但修改这个脚本重启会复原
需要禁用ipv6net这个服务,再手动配置/etc/network/interfaces

禁用ipv6net服务

systemctl stop ipv6net
systemctl disable ipv6net
reboot

手动配置ipv6地址(Ubuntu/Debian)

nano /etc/network/interfaces

添加如下

# 所有ipv6地址都不带中括号'['和']'
auto ipv6net
iface ipv6net inet6 v4tunnel
        address [your custom ipv6]
        netmask 64
        endpoint *.*.*.*
        local *.*.*.*
        ttl 255
        gateway [ipv6 gateway]

        up   /sbin/ip -6 addr add [another custom ipv6]/64 dev $IFACE
        up   /sbin/ip -6 addr add [yet another custom ipv6]/64 dev $IFACE

        down /sbin/ip -6 addr del [another custom ipv6]/64 dev $IFACE
        down /sbin/ip -6 addr del [yet another custom ipv6]/64 dev $IFACE

endpoint和local的ipv4地址根据搬瓦工后台的配置照抄,up和down的ipv6地址一一对应,复制多行可以添加多个自定义ipv6地址

# 注意事项

  • 现在(2025-08及以后)新注册的AWS账号没有永久免费的1TB/月下载流量,没有账号就不必折腾了

  • Cloudfront 可以根据不同的path设置不同的behavior，所以WS和XHTTP节点可以放在同一个CDN链接里共存

  • Cloudfront 有非常强大的分流功能,一个CDN资源可以添加多个源,可以是同域名不同端口/不同域名/甚至直连IP,还支持执行简单的脚本(Cloudfront Function的免费额度较少,Lambda@Edge按量计费,最好都别碰防被反薅).
    相比之下Cloudflare CDN免费版只支持重写端口.

  • 不一定要有自己的域名才能使用Cloudfront,Cloudfront只说要有一个域名DNS解析成ipv4地址(update 2025-12 ipv6 only也支持了),没说域名要是自己的或要有ssl证书,所以可以使用1.2.3.4.nip.io或fe80--1.sslip.io这类通配符DNS域名,自动把带ip地址的域名解析成对应的ip地址,作为CDN的源,实际上等于直接设置ip.

  • Cloudfront 上传按量收费，下载1TB以内免费，所以可以下行走Cloudfront上行走别的
    实测不主动上传文件,总流量200GB以内一般不会收费(会产生0.0x美元的账单但不会从卡里扣钱)
